# Удаление и безопасность при панике (Drop and Panic Safety)

Эй, вы заметили этот комментарий:

```rust
// Обратите внимание, что нам больше не нужно возиться с `take`,
// потому что всё является Copy и нет деструкторов, которые запустятся,
// если мы ошибемся... верно? :) Веееерно? :)))
```

Он правильный?

Извините, вы забыли, какую книгу читаете? Конечно же, он неправильный! (В каком-то смысле.)

Давайте еще раз посмотрим на внутреннее тело `pop_front`:

```rust ,ignore
// Возвращаем Box к жизни, чтобы мы могли забрать его значение и
// дропнуть его (Box продолжает магическим образом понимать это за нас).
let boxed_node = Box::from_raw(node.as_ptr());
let result = boxed_node.elem;

// Делаем следующий узел новым передним узлом.
self.front = boxed_node.back;
if let Some(new) = self.front {
    // Очищаем его ссылку на удаленный узел
    (*new.as_ptr()).front = None;
} else {
    // Если передний узел теперь null, то этот список теперь пуст!
    debug_assert!(self.len == 1);
    self.back = None;
}

self.len -= 1;
result
// Box неявно освобождается здесь и знает, что там нет T.
```

Вы видите баг? Как ни ужасно, это на самом деле вот эта строка:

```rust ,ignore
debug_assert!(self.len == 1);
```

*Серьезно*? Наша гребаная проверка целостности для тестов — это баг?? Да!!! Ну, если мы реализуем нашу коллекцию правильно, то она *не должна* им быть, но она может превратить что-то безобидное вроде «ой, мы плохо обновляем `len`» в *Эксплуатируемую Уязвимость Безопасности Памяти* (An Exploitable Memory Safety Bug)! Почему? Потому что она может вызвать панику! Большую часть времени вам не нужно думать или беспокоиться о паниках, но как только вы начинаете писать *действительно* небезопасный код и играть по правилам «инвариантов», вам нужно стать гипер-бдительными к паникам!

Нам нужно поговорить об [*исключительной безопасности* (exception safety)](https://doc.rust-lang.org/nightly/nomicon/exception-safety.html) (она же безопасность при панике, она же безопасность при раскрутке стека...).

Итак, вот в чем дело: по умолчанию паники являются *раскручивающими* (unwinding). Раскрутка — это просто причудливый способ сказать «заставить каждую функцию немедленно завершиться». Вы можете подумать: «Ну, если *все* завершают работу, то программа вот-вот умрет, так зачем об этом беспокоиться?», но вы ошибаетесь!

Мы должны заботиться об этом по двум причинам: деструкторы запускаются при возврате функции, и раскрутка может быть *перехвачена* (caught). В обоих случаях код может продолжить выполнение после паники, поэтому нам нужно быть очень осторожными и следить за тем, чтобы наши небезопасные коллекции всегда находились в *каком-то* согласованном состоянии всякий раз, когда может произойти паника, потому что каждая паника — это неявный ранний возврат!

Давайте подумаем, в каком состоянии находится наша коллекция, когда мы добираемся до этой строки:

У нас есть наш `boxed_node` в стеке, и мы извлекли из него элемент. Если бы мы вернулись в этот момент, `Box` был бы дропнут, а узел был бы освобожден. Теперь видите?.. `self.back` всё еще указывает на этот освобожденный узел! Как только мы реализуем остальную часть нашей коллекции и начнем использовать `self.back` для каких-то вещей, это может привести к использованию после освобождения (use-after-free)! Ой-ёй!

Интересно, что эта строка имеет аналогичные проблемы, но она гораздо безопаснее:

```rust ,ignore
self.len -= 1;
```

По умолчанию в отладочных сборках Rust проверяет наличие переполнений (underflows и overflows) и паникует, когда они происходят. Да, каждая арифметическая операция — это потенциальная угроза безопасности при панике! Эта операция *лучше*, потому что она происходит после того, как мы восстановили все наши инварианты, поэтому она не вызовет проблем с безопасностью памяти... до тех пор, пока мы не начнем слепо доверять тому, что `len` правильный, но если уж мы допустили отрицательное переполнение, то он определенно неправильный, так что мы в любом случае трупы! Строка с `debug_assert` в каком-то смысле *хуже*, потому что она может превратить незначительную проблему в критическую!

Я несколько раз упоминал термин «инварианты», и это потому, что это очень полезная концепция для безопасности при панике! По сути, для внешнего наблюдателя нашей коллекции существуют определенные свойства, которые мы всегда поддерживаем. Для `LinkedList` одно из них заключается в том, что любой узел, достижимый в нашем списке, всё еще выделен в памяти и инициализирован.

*Внутри* реализации у нас есть немного больше гибкости, чтобы *временно* нарушать инварианты, если мы не забудем восстановить их *прежде, чем кто-либо заметит*. На самом деле это одно из «киллер-фич» системы владения и заимствования Rust для коллекций: если операция требует `&mut Self`, то мы *гарантированно* имеем исключительный доступ к нашей коллекции, и для нас нормально временно нарушать инварианты, будучи уверенными, что никто не сможет скрытно в них влезть.

Возможно, величайшим выражением этого является [Vec::drain](https://doc.rust-lang.org/std/vec/struct.Vec.html#method.drain), который на самом деле позволяет вам полностью разрушить базовый инвариант `Vec` и начать перемещать значения из *начала* или даже *середины* `Vec`. Причина, по которой это *корректно* (sound), заключается в том, что возвращаемый нами итератор `Drain` удерживает `&mut` на `Vec`, и поэтому весь доступ заблокирован за ним! Никто не может наблюдать `Vec`, пока итератор `Drain` не исчезнет, и тогда его деструктор сможет «починить» `Vec` до того, как кто-либо заметит, это идеаль...

[Всё не так идеально](https://doc.rust-lang.org/nightly/nomicon/leaking.html#drain). К сожалению, вы [не можете полагаться на то, что деструкторы в коде, который вы не контролируете, обязательно запустятся](https://doc.rust-lang.org/std/mem/fn.forget.html), и поэтому даже с `Drain` нам нужно проделать немного дополнительной работы, чтобы наш тип всегда сохранял инварианты, но довольно забавным способом: [мы просто устанавливаем `len` у `Vec` в 0 в самом начале](https://doc.rust-lang.org/std/mem/fn.forget.html), поэтому если кто-то устроит утечку `Drain`, у него останется *безопасный* `Vec`..., но он также потеряет кучу данных. Ты утекаешь меня? Я утекаю тебя! Око за око! Настоящая справедливость!

Пример ситуации, где вы *действительно* можете использовать деструкторы для безопасности при панике, можно найти в [тематическом исследовании BinaryHeap::sift_up](https://doc.rust-lang.org/nightly/nomicon/exception-safety.html#binaryheapsift_up).

В любом случае, нам не понадобятся все эти навороты для наших `LinkedList`, нам просто нужно быть немного более бдительными в отношении того, где мы нарушаем наши инварианты, чему мы доверяем/требуем корректности, и избегать введения ненужных раскруток стека посреди сложных задач.

В данном случае у нас есть два варианта сделать наш код немного более надежным:

* Использовать такие операции, как `Option::take`, гораздо более агрессивно, потому что они более «транзакционны» и имеют тенденцию сохранять инварианты.

* Убить `debug_assert!` и довериться себе в написании более качественных тестов с выделенными функциями «проверки целостности», которые никогда не запустятся в пользовательском коде.

В принципе, мне нравится первый вариант, но он не очень хорошо работает для двусвязного списка, потому что всё закодировано с двойной избыточностью. `Option::take` не решил бы проблему здесь, но перемещение `debug_assert!` на строку ниже — решило бы. Но на самом деле, зачем усложнять себе жизнь? Давайте просто удалим эти `debug_assert!` и убедимся, что всё, что может вызвать панику, находится в самом начале или самом конце наших методов, где должно быть известно, что наши инварианты соблюдаются.

(В этом смысле, возможно, более точно думать о них как о *предусловиях* и *постусловиях*, но вы действительно должны стремиться относиться к ним как к инвариантам настолько, насколько это возможно!)

Вот наша полная реализация на данный момент:

```rust
use std::ptr::NonNull;
use std::marker::PhantomData;

pub struct LinkedList<T> {
    front: Link<T>,
    back: Link<T>,
    len: usize,
    _boo: PhantomData<T>,
}

type Link<T> = Option<NonNull<Node<T>>>;

struct Node<T> {
    front: Link<T>,
    back: Link<T>,
    elem: T, 
}

impl<T> LinkedList<T> {
    pub fn new() -> Self {
        Self {
            front: None,
            back: None,
            len: 0,
            _boo: PhantomData,
        }
    }

    pub fn push_front(&mut self, elem: T) {
        // SAFETY: это связанный список, чего вы хотите?
        unsafe {
            let new = NonNull::new_unchecked(Box::into_raw(Box::new(Node {
                front: None,
                back: None,
                elem,
            })));
            if let Some(old) = self.front {
                // Помещаем новый передний узел перед старым
                (*old.as_ptr()).front = Some(new);
                (*new.as_ptr()).back = Some(old);
            } else {
                // Если переднего узла нет, значит, список пуст, и нам нужно
                // установить также и задний узел.
                self.back = Some(new);
            }
            // Эти вещи происходят всегда!
            self.front = Some(new);
            self.len += 1;
        }
    }

    pub fn pop_front(&mut self) -> Option<T> {
        unsafe {
            // Нам нужно что-то делать только если есть передний узел для извлечения.
            self.front.map(|node| {
                // Возвращаем Box к жизни, чтобы мы могли забрать его значение и
                // дропнуть его (Box продолжает магическим образом понимать это за нас).
                let boxed_node = Box::from_raw(node.as_ptr());
                let result = boxed_node.elem;

                // Делаем следующий узел новым передним узлом.
                self.front = boxed_node.back;
                if let Some(new) = self.front {
                    // Очищаем его ссылку на удаленный узел
                    (*new.as_ptr()).front = None;
                } else {
                    // Если передний узел теперь null, то этот список теперь пуст!
                    self.back = None;
                }

                self.len -= 1;
                result
                // Box неявно освобождается здесь и знает, что там нет T.
            })
        }
    }

    pub fn len(&self) -> usize {
        self.len
    }
}
```

Что здесь может вызвать панику? Что ж, знание этого, честно говоря, требует от вас быть своего рода экспертом по Rust, но, к счастью, я им являюсь!

Единственные места в этом коде, которые, как я вижу, *возможно*, могут вызвать панику (за исключением какой-то абсолютной дичи, когда кто-то перекомпилирует `stdlib` с включенными `debug_assert!`, но этого делать никогда не стоит) — это `Box::new` (при нехватке памяти) и арифметика `len`. Всё это находится в самом конце или самом начале наших методов, так что да, мы ведем себя мило и безопасно!

...вы были удивлены тем, что `Box::new` может вызвать панику? Паники подстерегают вас именно так! Старайтесь сохранять эти инварианты, чтобы вам не пришлось об этом беспокоиться!
