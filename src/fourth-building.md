# Построение (Building Up)

Хорошо, начнем с построения списка. С этой новой системой это довольно просто.
`new` по-прежнему тривиален, просто устанавливает все поля в `None`.

Теперь давайте попробуем написать добавление в начало списка (`push_front`). Поскольку
двусвязные списки значительно сложнее, нам придется
проделать изрядный объем работы. Там, где операции с односвязным списком могли быть
сведены к простой однострочной функции, операции с двусвязным списком довольно сложны.

В частности, теперь нам нужно специально обрабатывать некоторые граничные случаи вокруг
пустых списков. Большинство операций будут затрагивать только указатель `head` или `tail`.
Однако при переходе к пустому списку или от него нам нужно изменять
*оба* указателя одновременно.

Простой способ проверить, имеют ли смысл наши методы, — это поддерживать
следующий инвариант: на каждый узел должно быть ровно два указателя.
На каждый узел в середине списка указывают его предшественник и
преемник, в то время как на узлы на концах указывает сам список.

Давайте попробуем:

```rust ,ignore
pub fn push_front(&mut self, elem: T) {
    // новому узлу нужно +2 ссылки, всё остальное должно быть +0
    let new_head = Node::new(elem);
    match self.head.take() {
        Some(old_head) => {
            // непустой список, нужно подключить old_head
            old_head.prev = Some(new_head.clone()); // +1 new_head
            new_head.next = Some(old_head);         // +1 old_head
            self.head = Some(new_head);             // +1 new_head, -1 old_head
            // итого: +2 new_head, +0 old_head -- ОК!
        }
        None => {
            // пустой список, нужно установить tail
            self.tail = Some(new_head.clone());     // +1 new_head
            self.head = Some(new_head);             // +1 new_head
            // итого: +2 new_head -- ОК!
        }
    }
}
```

Почему мы не можем получить доступ к полям `prev` и `next` наших узлов? Это работало раньше,
когда у нас был просто `Rc<Node>`. Похоже, `RefCell` мешает.

> Общие изменяемые контейнеры (Shareable mutable containers).
>
> Значения типов `Cell<T>` и `RefCell<T>` могут быть изменены через разделяемые ссылки (т.е.
> обычный тип `&T`), тогда как большинство типов Rust могут быть изменены только через уникальные (`&mut T`)
> ссылки. Мы говорим, что `Cell<T>` и `RefCell<T>` предоставляют «внутреннюю изменяемость» (interior mutability), в отличие
> от типичных типов Rust, которые проявляют «унаследованную изменяемость» (inherited mutability).
>
> Типы ячеек бывают двух видов: `Cell<T>` и `RefCell<T>`. `Cell<T>` предоставляет методы `get` и `set`,
> которые изменяют внутреннее значение за один вызов метода. Однако `Cell<T>` совместим только
> с типами, реализующими `Copy`. Для других типов необходимо использовать тип `RefCell<T>`,
> приобретая блокировку на запись перед изменением.
>
> `RefCell<T>` использует времена жизни Rust для реализации «динамического заимствования», процесса, посредством которого можно
> заявить о временном эксклюзивном изменяемом доступе к внутреннему значению. Заимствования для `RefCell<T>`
> отслеживаются «во время выполнения» (at runtime), в отличие от нативных типов ссылок Rust, которые полностью отслеживаются
> статически, во время компиляции. Поскольку заимствования `RefCell<T>` являются динамическими, возможна попытка
> заимствовать значение, которое уже заимствовано как изменяемое; когда это происходит, результатом является паника потока.
>
> # Когда выбирать внутреннюю изменяемость
>
> Более распространенная унаследованная изменяемость, когда для изменения значения необходимо иметь уникальный доступ, является
> одним из ключевых элементов языка, который позволяет Rust строго рассуждать об алиасинге указателей,
> статически предотвращая баги с падениями. Из-за этого предпочтительнее унаследованная изменяемость, а
> внутренняя изменяемость является чем-то вроде крайней меры. Поскольку типы ячеек позволяют производить изменения там, где это
> иначе было бы запрещено, бывают случаи, когда внутренняя изменяемость может быть уместной или даже *должна* использоваться, например:
>
> * Введение корней унаследованной изменяемости в разделяемые типы.
> * Детали реализации логически неизменяемых методов.
> * Изменяемые реализации `Clone`.
>
> ## Введение корней унаследованной изменяемости в разделяемые типы
>
> Разделяемые типы умных указателей, включая `Rc<T>` и `Arc<T>`, предоставляют контейнеры, которые могут быть
> склонированы и разделены между несколькими сторонами. Поскольку содержащиеся в них значения могут иметь
> несколько алиасов, они могут быть заимствованы только как разделяемые ссылки, но не как изменяемые ссылки.
> Без ячеек было бы вообще невозможно изменять данные внутри разделяемых боксов!
>
> Поэтому очень часто `RefCell<T>` помещают внутрь типов разделяемых указателей, чтобы вернуть
> изменяемость:
>
> ```rust ,ignore
> use std::collections::HashMap;
> use std::cell::RefCell;
> use std::rc::Rc;
>
> fn main() {
>     let shared_map: Rc<RefCell<_>> = Rc::new(RefCell::new(HashMap::new()));
>     shared_map.borrow_mut().insert("africa", 92388);
>     shared_map.borrow_mut().insert("kyoto", 11837);
>     shared_map.borrow_mut().insert("piccadilly", 11826);
>     shared_map.borrow_mut().insert("marbles", 38);
> }
> ```
>
> Обратите внимание, что в этом примере используется `Rc<T>`, а не `Arc<T>`. `RefCell<T>` предназначены для
> однопоточных сценариев. Рассмотрите возможность использования `Mutex<T>`, если вам нужна разделяемая изменяемость в
> многопоточной ситуации.

Эй, документация Rust продолжает быть невероятно крутой.

Самая суть, которая нас интересует, — это вот эта строка:

```rust ,ignore
shared_map.borrow_mut().insert("africa", 92388);
```

В частности, штука `borrow_mut`. Похоже, нам нужно явно заимствовать
`RefCell`. Оператор `.` не сделает этого за нас. Странно. Давайте попробуем:

```rust ,ignore
pub fn push_front(&mut self, elem: T) {
    let new_head = Node::new(elem);
    match self.head.take() {
        Some(old_head) => {
            old_head.borrow_mut().prev = Some(new_head.clone());
            new_head.borrow_mut().next = Some(old_head);
            self.head = Some(new_head);
        }
        None => {
            self.tail = Some(new_head.clone());
            self.head = Some(new_head);
        }
    }
}
```
